01 — Visão Geral

O que é um seguidor de linha?

Um robô seguidor de linha é um sistema autônomo capaz de detectar e percorrer uma trajetória marcada no chão — geralmente uma faixa preta sobre fundo branco — sem intervenção humana. É o "Olá Mundo" da robótica autônoma: simples o suficiente para aprender em dias, mas complexo o suficiente para ensinar os fundamentos reais do trabalho em equipe de engenharia.

No Desafio de Nivelamento 2026.1, vocês precisam projetar, simular e documentar um seguidor de linha em grupo, cobrindo quatro frentes: Gestão, Mecânica, Eletrônica e Programação. Este artigo vai destrinchar cada uma delas.

✅ Boa notícia

A entrega física não é obrigatória. Simulações em Tinkercad, Webots, Fusion 360 ou qualquer software são totalmente aceitas. Foque na qualidade do relatório.

Como o robô "enxerga" a linha?

A magia toda está no sensor infravermelho (IR). Ele emite luz IR e mede o quanto é refletido de volta: superfície branca reflete muito (sinal alto), superfície preta absorve quase tudo (sinal baixo). Com dois ou mais sensores posicionados embaixo do chassi, o Arduino consegue saber se o robô está sobre a linha, saindo pela esquerda ou saindo pela direita.

1
Sensor lê a pista
Os sensores IR apontam para o chão e retornam valores digitais (0 ou 1) indicando se estão sobre linha preta ou fundo branco.
2
Arduino processa
O microcontrolador lê os pinos de entrada, compara os estados dos sensores e calcula o erro de posição em relação à linha.
3
Motores corrigem
Via Ponte H (L298N), o Arduino ajusta a velocidade de cada motor para curvar o robô de volta à linha.

⚠️ Atenção: GND Comum
O GND da bateria, do L298N e do Arduino DEVEM estar todos conectados no mesmo ponto (GND comum). Sem isso, o circuito não funciona.

Por que usar Ponte H (L298N)?

Motores DC consomem corrente demais para serem ligados diretamente nos pinos do Arduino (que suportam no máximo ~40mA). A Ponte H é um circuito de potência que deixa o Arduino controlar sem alimentar diretamente os motores. Ela também permite inverter o sentido de rotação (necessário para curvas e ré).

O controle de velocidade é feito via PWM (Pulse Width Modulation): o Arduino "pulsa" o sinal em alta frequência e a variação da largura do pulso define a velocidade média do motor.

04 — Estrutura e Mecânica

Chassi e posicionamento

O chassi é a espinha dorsal do robô. Para o Desafio, você pode projetar em Fusion 360, Tinkercad 3D, ou até desenhar à mão com cotações. O importante é que o projeto seja justificado.

Configuração de acionamento: diferencial

A configuração mais simples — e recomendada — é a acionamento diferencial: dois motores nas laterais traseiras + uma roda boba na frente. Para virar, basta aplicar velocidades diferentes nos dois motores.

📐 Altura dos sensores

Mantenha os sensores IR entre 3mm e 10mm do chão. Muito alto → leitura imprecisa. Muito baixo → pode raspar na pista. Sensor na linha central entre as rodas funciona bem como ponto de partida.

Material do chassi

A
Acrílico cortado a laser
Profissional, leve e fácil de furar. Precisa de acesso a uma cortadora. Excelente para portfólio.
B
MDF (6mm)
Barato, fácil de trabalhar com ferramentas básicas. Pode ser cortado com estilete e furadeira. Boa opção para prototipagem.
C
Impressão 3D (PLA)
Máxima liberdade geométrica. Se o grupo tem acesso a impressora 3D, é a melhor escolha para demonstrar habilidade em CAD.
05 — Programação

Lógica e código C/C++

O código é o coração do robô. Vamos do mais simples ao mais sofisticado, para vocês escolherem o nível certo para o relatório.

Nível 1: Controle Bang-Bang (Liga/Desliga)

A abordagem mais direta: se o sensor esquerdo detectar linha, vira à esquerda; se for o direito, vira à direita. Funciona, mas causa um movimento ziguezagueante.

C/C++ — Arduino seguidor_bangbang.ino 
// ============================================
// GAIA Robótica — Seguidor de Linha Bang-Bang
// Trainee 2026.1
// ============================================

// Pinos dos sensores IR (0 = linha, 1 = branco)
const int SENSOR_ESQ = A0;
const int SENSOR_DIR = A1;

// Pinos de controle dos motores (via L298N)
const int ENA = 5;   // PWM Motor A (esquerdo)
const int IN1 = 6;
const int IN2 = 7;
const int ENB = 10;  // PWM Motor B (direito)
const int IN3 = 8;
const int IN4 = 9;

const int VEL_BASE = 180;  // 0-255

void setup() {
  pinMode(SENSOR_ESQ, INPUT);
  pinMode(SENSOR_DIR, INPUT);
  pinMode(ENA, OUTPUT); pinMode(IN1, OUTPUT); pinMode(IN2, OUTPUT);
  pinMode(ENB, OUTPUT); pinMode(IN3, OUTPUT); pinMode(IN4, OUTPUT);
}

// Funções auxiliares de movimento
void frente(int vel) {
  analogWrite(ENA, vel); analogWrite(ENB, vel);
  digitalWrite(IN1, HIGH); digitalWrite(IN2, LOW);  // Motor A frente
  digitalWrite(IN3, HIGH); digitalWrite(IN4, LOW);  // Motor B frente
}

void viraEsq() {
  analogWrite(ENA, VEL_BASE / 2);  // Reduz motor esquerdo
  analogWrite(ENB, VEL_BASE);
  digitalWrite(IN1, HIGH); digitalWrite(IN2, LOW);
  digitalWrite(IN3, HIGH); digitalWrite(IN4, LOW);
}

void viraDir() {
  analogWrite(ENA, VEL_BASE);
  analogWrite(ENB, VEL_BASE / 2);  // Reduz motor direito
  digitalWrite(IN1, HIGH); digitalWrite(IN2, LOW);
  digitalWrite(IN3, HIGH); digitalWrite(IN4, LOW);
}

void loop() {
  int esq = digitalRead(SENSOR_ESQ);  // 0 = linha
  int dir = digitalRead(SENSOR_DIR);

  if (!esq && !dir)  frente(VEL_BASE);   // Ambos na linha → reto
  else if (!esq && dir)  viraEsq();      // Só esquerdo → vira esq
  else if (esq && !dir)  viraDir();      // Só direito → vira dir
  else               frente(VEL_BASE);   // Nenhum → continua
}

Nível 2: Controle Proporcional (P)

Usando sensores analógicos em vez de digitais, conseguimos calcular um erro contínuo e corrigir os motores proporcionalmente — o robô curvará suave quando estiver levemente desviado e bruscamente quando estiver muito fora.

Referência
Posição ideal (centro)
→
Erro
ref − leitura atual
→
Controlador P
correção = Kp × erro
→
Motores
vel_esq ± correção
→
Robô na linha
erro → 0
C/C++ — Arduino seguidor_proporcional.ino 
// Seguidor com controle Proporcional (P)
// Usa sensores analógicos para erro contínuo

const float Kp = 0.5;   // Constante proporcional — AJUSTE isto!
const int   VEL_BASE = 160;

void loop() {
  int leitura_esq = analogRead(SENSOR_ESQ);  // 0-1023
  int leitura_dir = analogRead(SENSOR_DIR);

  // Erro: positivo = desviou pra direita, negativo = pra esquerda
  int erro = leitura_dir - leitura_esq;

  int correcao = (int)(Kp * erro);

  int vel_esq = constrain(VEL_BASE + correcao, 0, 255);
  int vel_dir = constrain(VEL_BASE - correcao, 0, 255);

  moverMotores(vel_esq, vel_dir);
}
🚀 Diferencial no relatório

Para pontuação extra em Criatividade e Resolução de Problemas, implemente ou ao menos descreva um controlador PID completo. Explique o que é a parcela integrativa (I) e derivativa (D) — o artigo sobre Controle PID aqui no site cobre isso em detalhe.

06 — Gestão do Projeto

Organização e cronograma

Com apenas 4 dias entre início e entrega (02/10 a 05/10), a gestão eficiente é o que separa grupos que entregam dos que ficam com um documento incompleto às 23h50 do prazo.

Sugestão de divisão de tarefas

D1
Kickoff e divisão (02/10)
Definir quem faz cada frente (Gestão, Mecânica, Eletrônica, Programação). Abrir o documento do relatório com a estrutura já montada. Cada responsável começa sua seção.
D2
Desenvolvimento (03/10)
Mecânica: CAD do chassi. Eletrônica: esquemático no Tinkercad. Programação: escrever e comentar o código. Gestão: lista de materiais + cronograma detalhado.
D3
Integração e simulação (04/10)
Juntar todas as partes no relatório. Rodar simulações e capturar prints/gifs. Revisar o texto e verificar se todos os tópicos obrigatórios estão cobertos.
D4
Revisão final e envio (05/10)
Revisar formatação, capa com nomes/RA, exportar PDF. Enviar para gaia.robotica.uftm@gmail.com antes das 23h59.
⚠️ Atenção ao prazo

O não envio dentro do prazo implica desclassificação. Não deixem para a última hora. Enviem uma versão "boa o suficiente" às 22h e melhorem até 23h30 se necessário.

07 — Relatório Técnico

Como estruturar o documento

O relatório é o produto final avaliado. Segue um checklist baseado nos critérios obrigatórios definidos no edital do desafio:

✓
Capa
Nome completo e RA de todos os integrantes, título do projeto, data. Simples, mas eliminatório se faltar.
✓
Introdução
O que é um seguidor de linha? Como funciona o sensor IR? Qual a aplicação real desse tipo de robô? Cite exemplos industriais (AGVs em fábricas, por exemplo).
✓
Desenvolvimento
Esquema eletrônico (print do Tinkercad), concepção mecânica (render/foto do CAD), lista de materiais com custos, dificuldades encontradas e como foram resolvidas.
✓
Conclusão
Código comentado completo, prints/gifs da simulação funcionando, propostas de melhoria (ex.: adicionar PID completo, mais sensores, Bluetooth para tuning remoto). Se houver artistas no grupo: protótipos visuais do robô.

Dicas de escrita técnica

✅ Faça

Explique o porquê de cada decisão. "Escolhemos o L298N pois suporta até 2A por canal, suficiente para os motores utilizados."

❌ Evite

Descrições genéricas sem justificativa. "Usamos o L298N porque é um componente comum em projetos de robótica."

✅ Faça

Documente os problemas reais que tiveram. "O sensor direito retornava ruído excessivo — adicionamos um capacitor de 100nF para filtrar."

❌ Evite

Fingir que tudo correu perfeito. Avaliadores sabem que projetos sempre têm problemas. Honestidade técnica é um diferencial.

08 — Indo além

Melhorias e próximos passos

Se você quer se destacar — ou simplesmente tem curiosidade — aqui estão ideias de melhorias que podem ser descritas na seção de conclusão do relatório:

🚀 Ideias de melhoria

+ Sensores: Usar 5 sensores em vez de 2 aumenta muito a precisão da detecção.

PID completo: Implementar as parcelas I e D para eliminar o oscilação no Bang-Bang.

Bluetooth: Adicionar módulo HC-05 para ajustar Kp/Ki/Kd via serial sem precisar regravar o firmware.

Sensor de fim de pista: Detector de quando não há mais linha (robô saiu da pista) para parar os motores automaticamente.
